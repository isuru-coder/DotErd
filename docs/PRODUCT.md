# D400.DotErd Product Plan

## Product Summary

D400.DotErd is a developer-focused .NET command line tool that reads the relational model represented by Entity Framework Core `DbContext` classes and generates or updates a native editable draw.io ERD.

Phase 1 focuses on code-first EF Core projects using SQL Server relational mappings. It does not connect to production databases and does not infer relationships beyond the EF Core model.

## Target Users

- .NET developers maintaining EF Core code-first applications.
- Technical leads reviewing schema changes in pull requests.
- Architects who need editable ERDs that stay aligned with source-controlled models.
- Teams that want schema snapshots and diff reports without adding a web service.

## Phase 1 Value Proposition

D400.DotErd should let a developer run a CLI command against one or more `DbContext` classes and receive:

- A native draw.io XML diagram that can be edited in diagrams.net.
- Preserved table positions when updating an existing diagram.
- A JSON snapshot of the extracted relational schema.
- A schema difference report between snapshots.
- A verification command suitable for CI.

## Supported Inputs

- .NET projects containing Entity Framework Core code-first models.
- One or multiple `DbContext` classes.
- SQL Server provider metadata and relational mappings.
- Existing draw.io XML files generated by D400.DotErd.
- Existing schema snapshot JSON files generated by D400.DotErd.

## Supported Model Features

- Schemas and tables.
- Columns and SQL data types.
- Required and nullable columns.
- Primary keys and composite keys.
- Foreign keys.
- One-to-one and one-to-many relationships.
- Many-to-many join tables as represented by EF Core relational metadata.
- Basic indexes.

## Explicit Non-Goals

- UML diagrams.
- Database providers other than SQL Server.
- Production database connections.
- Web UI.
- AI relationship inference.
- Stored procedure diagrams.

## Primary Workflows

### Generate a New ERD

A developer points the CLI at a project and one or more `DbContext` classes. The tool builds or loads the EF Core model, extracts the relational schema, writes a snapshot, and creates a draw.io diagram.

### Update an Existing ERD

A developer runs the same command against an existing `.drawio` file. The tool preserves known table positions, updates table contents and relationships, adds new tables, and removes or marks obsolete diagram elements according to the selected behavior.

### Review Schema Changes

A developer compares two schema snapshots and receives a deterministic difference report covering tables, columns, keys, foreign keys, and indexes.

### Verify Generated Artifacts

CI runs a verification command that regenerates expected outputs in memory or a temporary location and fails if committed diagram, snapshot, or report outputs are stale.

## Public CLI Surface

The exact command names are unresolved, but Phase 1 should expose these capabilities:

```text
d400 doterd generate
d400 doterd diff
d400 doterd verify
```

Required command capabilities:

- Select project or assembly containing the EF Core model.
- Select one or more `DbContext` classes.
- Choose output draw.io file path.
- Choose output snapshot path.
- Read an existing draw.io file for position preservation.
- Read old and new snapshots for diff reports.
- Produce machine-readable and human-readable results where useful.
- Return stable exit codes for automation.

## Success Metrics

- Developers can generate an editable ERD from a representative SQL Server EF Core code-first project without manual XML editing.
- Re-running generation without model changes produces no meaningful file changes.
- Existing table positions survive diagram updates.
- Snapshot diffs are deterministic and reviewable.
- Verification mode can be used in CI.

## Key Product Risks

- Loading arbitrary EF Core projects may require build-time context that is hard to reproduce from a generic CLI.
- Design-time `DbContext` creation can fail when projects depend on application configuration, secrets, or service registration.
- EF Core metadata has provider-specific details that must be handled carefully for SQL Server type fidelity.
- draw.io XML must remain native and editable, not just visually correct.
- Auto layout quality can quickly become subjective; Phase 1 should prefer predictable over beautiful.

## Assumptions

- Phase 1 users can build their EF Core project locally.
- The CLI can use EF Core design-time conventions or an explicit design-time factory when available.
- The draw.io file is source-controlled alongside the application.
- Snapshot JSON is a first-class artifact, not an internal cache.
